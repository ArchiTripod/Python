首先要说明的是，代码不是小说和其它文字作品，代码不能「读」。我们平时所谓的「读」代码其实是 explore 或者 decypher。

所以读代码有很强的目的性，也需要有很多先决条件。第一个先决条件是知道这个 code base 的用途和用法。然后要了解工作的基本原理。最好还要有 high-level design 的文档。

目的性可以有两种：第一是扩展 code base 增加原本没有的功能。这个分为以下几步：
  1. 首先要假想系统是如何工作的。这种假象要着重在为什么所需的功能现在没有提供，如果实现的话大体难度是什么。
  2. 阅读代码部分验证或者推翻原来的想法。这种阅读要结合 debugging。
  3. 建立新的假想，重复第二步。
  4. 如果假想与所看的 code 基本吻合，尝试修改 code。
  5. 测试。如果测试不能通过，退回到第四部甚至第二部。

第二种是验证某些想法。这种阅读是上面过程的第 1-3 步。这种过程中的假想可能是一个归谬的假象——也就是说，根据系统的 high-level design，推断出某个功能无法实现，但是现实中的 code base 提供了这个功能。需要通过 explore code 来解释假想为什么是错误的。

总之，代码是不可能像小说那样去「读」的。在 explore 之前就必须做好足够的功课。我阅读代码从来不会在没有「假想」的情况下无目的的瞎看。Explore 代码和物理实验有类似之处，没有经过设计的实验时没有收获的。

谢科，这一秒钟正在Twitter写代码

收起
Jason Chung、fans656、蔡少骏 等人赞同
(既然大家都想和我搞基，那就公布github id吧: oeddyo (Eddie Xie) · GitHub )

楼上冯东说得很好，“读”代码是不能给你带来任何收益的，正如“读书”一样，如果在读的时候你不琢磨，保管你读完仨月准忘了一大半。真正需要的是去“试”代码，动手去调调代码，改改这改改那，看看把A变成B这个代码的结果会有什么变化。

既然他讲了道理，那我就来推荐一个切实可操作且已经被证明（被我）有成效的方法吧：直接去github上（咱就假设已经开源了吧），找找它的issues。

举个例子，你对大数据处理感兴趣，你听说了Twitter出品了一个牛逼闪闪的写map-reduce的工具，搜到了它的github首页 (twitter/scalding · GitHub)


然后看到右边的"issues"了么？点它，你就可以看见现在这个项目里，还有哪些问题可以/需要解决。

为什么一定要找问题？记不记得当初高中英文老师告诉你：阅读理解要先看问题，带着问题去找答案。这个不是没有根据的，读源代码也是一样，如果你是漫不经心地读，那么给你打包票，没有几个人能把它给“读”下来，因为通常的代码都太大太大了，现代计算机已经发展到了即使是一个小小的软件包，也够你用三五年的时间才能深入到明白前因后果，明白里面的每个细节。

这给试图“读”代码的初学者带来了极大的困难（我当初也纳闷这问题，而且我还问过这问题！只不过当时没知乎我就问的边上的人罢了）。当你想拿起linux的代码来读，刚把内核读了一点点，前面刚明白后面又忘了吧。况且，仅“读”的时候，会给你一种错觉——让你感觉你啥都读到了一点，真要你来动动改改的话，保管没法动手。

那么这时候，去github解issue的好处就体现出来了。

拿这个项目来说，你点了issue (Issues · twitter/scalding · GitHub)之后会出现:

有没有看到那个beginner? 猛戳呀！意思就是，这个项目的开发者认为，这个issue可以由你这样的初学者来解决。

凭啥你要帮他写代码作嫁裳呢？关于贡献开源软件的哲学争辩我就不多说了，简单讲讲益处有几个
  1. 你可以真正地通过解这个issue来帮助你自己了解这代码是在干嘛
  2. 它给你一个跟世界上最顶尖的程序员交流的机会（为毛？因为你写完了人家要给你review呀亲！）而且这是免费的哦！
  3. 顺便，你的贡献还可以当作你最不可争议的简历

2和3就不多说了，不切题，属于其它益处，说说1吧。

为啥解个issue可以帮你“读”代码？

简单，因为一个issue足够小，而且多数属于可分割的小任务（尤其是标了个beginner的那种，快抢啊！）。我就不引用研究了，你自己的体验应该也会告诉你：当你在试图解决手里的一个小任务，而不是被一个大任务吓到的时候，你会更专注，更容易下手。

那么好，你开始解这个issue了。为了解决一个小问题，你至少得知道这个小问题是由啥引起的吧？这样就引导着你“带着问题读文章”了。相信我，这个时候读的效率，远高于吊儿郎当左翻右翻。

读着读着，发现卡住了。咋办？

解issue的第三个好处又来了，既然你在热心地帮人家解决问题，要是你在试图解决问题的这个过程中遇到了问题，人家肯定也愿意来帮你解决这个因为试图解决问题而遇到的问题呀！（嗯我故意的）。我的经验表明，只要你谦虚好学，牛逼大牛也肯跳出来教你

第四个，解issue可不光只是读代码哦，你要改代码。改的过程中自然而然你得跑test吧，test不过你又得改吧？这叫啥？这叫正反馈啊亲！而且这样的正反馈来得极其迅速，差不多你改一点，五分钟内你知道你改得对错。读代码的时候有这效果吗？没有呀，回忆一下上一次，你读完了一段二分，觉得明白了，没去写写。面试的时候去写，大汗淋漓，写了半天写不对（编程珠矶里的故事啦），为啥？因为你的大脑骗了你，它觉得你肤浅（superficial)地以为自己懂了，其实你还没懂它就已经累了要求你跳过了。

而有正反馈的机制就不一样了，我才不信一片compliation error的时候你的大脑能骗你test过了呢，哼！

第五个，成就感。这个不用我说吧，就像打网游打怪一样，你解了多少issue，猎头姐姐是看得见的哟！下次没准就约你喝咖啡去了。


好，总结一下吧：
个人觉得最有益的读源码方式是去帮那个源码项目解issue（当然它如果是个内部项目，那你就内部解嘛，不一定要到基友平台github上啦）。而因为解issue的特性，它是我目前感觉到的，最快迅的深入了解一个library的方法。

非要credibility吗？好吧，仨月前其实我只是上面那个project的用户，解着解着issue现在也算排名靠前的贡献者了吧。欢迎来github上follow我，给我加星啥的，一起搞基啥的，萌萌哒

程序员会需要经常"读"代码，不过目的各有不同。


即使你是学习，也要带目的去读，最好记一下笔记。简单的把源码全看一遍，很快会忘记。而且太耗时间。(如果代码量大的话)


如果你想学网络编程，就看网络层的代码，看如何发请求，如何接收返回，如何解析数据这一块，其他的暂时不要看。你想学UI，就看如何编写UI的代码，这样一块一块的学习下去，很快就能搞清楚全套源码的架构和功能了。


如果是学习目的，千万不要试图读懂一些规模很大的开源代码，如Linux之类，只关注你真的需要的某个模块好了。在公司里，研究大的开源代码，是需要相当多人手一起合作才能完成的。

看源码一般就是好奇心或者被逼的

好奇心：
  1. 这个做出来的功能好屌，怎么实现的，我去瞅一瞅
  2. 这个很出名啊，我去看看它的代码是不是很屎，能不能喷一下
  3. 这个人好屌啊，我去看看他的代码是不是很屎，能不能喷一下
被逼的：能力不及或者时间不够或者懒的情况下开发一个功能，用了一个别人的库，它文档又不全，有个小功能又需要它的一个特性，没办法，只能扒代码了

阅读源码可以先用一下它，从它的功能入手，看着看着，就会有一种很high的感觉『哦！原来是这么搞的！』

收起
木易、徐志刚、碎叶之城 等人赞同
最近正好一直在看一些开源项目的源代码有几点感想吧。
  • 看一些优秀的源代码能够增强你写代码的美感，比如：（好处真的很多）
    • 变量的命名
    • 代码的书写方式
    • 模块的组织方式
  • 有一些开源项目的API说明是比较缺乏的，可能说的不够明白或者没有把所有API列举出来。

比如我最近想实现一个功能，找遍了API文档都没有找到答案，可是直觉上觉得肯定已经有这个功能，最后在源代码中找到了一个内部方法来解决。（还顺便给他们修复了一个BUG）

关于 @赖勇浩提到的“不读懂的开源项目不敢用” 这个观点，我的想法是要区分这个开源项目是否是属于你项目最核心的代码部分吧。如果你说你用一个弹出框的插件都要去读他们的源代码，那就真的没有必要了。可是如果那个开源项目属于你们骨架级别的，我觉得不需要全部读懂，可是起码要了解框架背后的运行逻辑。不要完全一头瞎。

如果阅读源代码的话。我也在学习当中，没有什么太好的经验。不过有几个思路可以提供一下：（主要针对前端项目），希望有大牛提供更好的思路和想法。
  1. 和他们的作者联系一下，询问一下如果你想深入了解的话，最好怎么去做，相信我，一般来说会让你得到惊喜的
  2. 看项目的打包程序（grunt、gulp）等的任务上面打包顺序，依次大概浏览一下
  3. 看命名，一般都会有什么core.js之类的一看就知道比较核心的代码文件，最好深入看

暂时只想到这些，后续有新想法再补充。

那么阅读源代码的正确打开方式是什么呢，我可以不负责任地告诉你是“文档”，不是让你去看文档，而是让你千万别看文档。你可以去看看一些死得早或者是比较老的项目C或者CPP语言项目，比如v8吧，最直接的API文档就是自个儿去看v8.h，当然也有好心人翻译成文档了最后。

我的意思呢，是能不看文档就别看文档，去直接翻代码。做项目遇到问题了，首先不要看文档是怎么写的，先看你调用的那个函数或者引用的什么变量，是怎么个内部实现的，为什么会出现你的那个bug，是你自己没用对呢，还是这里是框架内部的bug。这样你看代码就是带着目的去看，这样你阅读代码的效果绝对地事半功倍，等你找到了问题，如果发现是框架自己有问题，还能给作者提个PR或者写个邮件，提交你的patch，不仅读代码能力锻炼了，同时也锻炼了交流沟通能力，碰上个老外写的项目，还能发几封英文邮件装一装。

当然，上一阶段必须建立在你可以轻松根据源码去定位你所遇到的问题，其实说真的，要到这一步，你还真得有一定的代码阅读量（这是必然的），不过不必担心，第一次阅读源码的过程大致只是熟悉下项目本身的布局，比如哪个文件是干什么的，很多时候上手了看文件名就能摸清楚个大概。等到你具体要修bug的时候，再仔细阅读某一个部分，不过最重要的一点还是：先让你的代码跑起来，边调试边看。

好了，对于楼主说的体验，其实也没什么，就是原来会抱怨文档里写得不全，然后各种Google才得出结论，还可能会模棱两可。现在呢，会看源码之后，减少了使用搜索引擎的次数，因为直接可以从代码里面找到答案，有时候也会使用作者文档里没有expose的接口（不推荐），并且由内容消费者转变为内容产生者（千万要记得100%分享你的阅读经验给其他还不能很好阅读源代码，在使用搜索引擎的初学者）。

源码是什么？是本质
文档再好，不过是提纲目录，做得再饱满，也是对源码的补充说明，为了更容易懂、更方便用

而源码里能够看到的，不仅仅是功能模块的解决方案，更是作者对于问题的解决方案、对于事物的理解认知、对于世界的观点理念

所以说，程序员阅读源码是一种什么心态？解决问题 + 理解事物，更重要的：看到源码背后的想法

In Coding We Trust. 在Coding的世界中，阅读源码对编程的意义，就像是读书对人生的意义一样，有欢喜、有吐槽、有感悟、有鄙夷、有启发、有无奈……

古人云：读万卷书，行万里路；
程序员说：读万行源码，行万种实践；

至于如何才能更好阅读代码？推荐一种方法，也不尽然适合所有人：
  • 判断自己能够hold住的系统复杂度在什么级别，单个文件、某种设计模式、一个子模块、一个子系统、一个复杂主系统、多系统混合业务、大规模综合系统等
  • 将问题拆解到自己能够hold住的级别，目的是避免一口吃个胖子，量力而行
  • 搞清楚代码结构，自己画出来类图、架构图，目的是整个架构了然于胸
  • 配合文档、注释，串通主流程，其他细节就可以在问题追溯中进行了
